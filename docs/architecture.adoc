== 1. Overview

The Dental Office Voice Receptionist is a voice-first appointment booking agent for a dental practice. Callers interact with the agent through spoken natural language — the system handles greeting, language selection, patient lookup, slot proposal, booking confirmation, and graceful escalation to a human receptionist.

*Phase 0 (this repo)* is a proof-of-concept that runs entirely on localhost. A browser-based WebRTC client (`static/index.html`) replaces a real telephone connection. No external telephony account or cloud subscription is required — the only optional external dependency is the Groq API for LLM inference; the system also runs fully offline with Ollama.

*Supported locales:* The agent speaks German (formal _Sie_-form) or English. The active locale is set once at startup via the `OFFICE_LOCALE` environment variable (`de` or anything else → `en`). Language is locked for the entire call — there is no mid-call detection. The initial greeting, TTS voice, and after-hours routing message are all derived from this value.

The pipeline is built on *Pipecat 1.0.0* with the `pipecat-flows` library managing the conversation state machine. The LLM backend is configurable: either *Groq* (`llama-3.3-70b-versatile`, cloud) or *Ollama* (`qwen2.5:14b`, local). Both are exposed through the OpenAI-compatible `OpenAILLMService`. Speech recognition uses *faster-whisper* (multilingual EN/DE). Text-to-speech uses *Piper* with locally cached `.onnx` model files.

== 4. Conversation State Machine

The flow is managed by *pipecat-flows* `FlowManager`. Each state is a `NodeConfig` with a set of callable functions; state transitions happen exclusively via tool call return values from the handlers in `flows/nodes.py`.

[plantuml,state-machine,svg]
....
@startuml
skinparam backgroundColor #FEFEFE
skinparam StateBackgroundColor #EEF4FF
skinparam StateBorderColor #4466AA

[*] --> consent : on_client_connected\n(transport event)\n\npre_actions [tts_say]:\n  1. greeting\n  2. consent notice

consent --> hours_check : record_consent(given=true)\n[logs consent event]
consent --> handoff : record_consent(given=false)\n[reason: consent_declined]

hours_check --> intent : get_office_hours()\nreturns open\n[global fn]
hours_check --> closing : get_office_hours()\nreturns closed

intent --> collect_info : set_intent("booking")
intent --> manage_appointment : set_intent("reschedule"\n | "cancel")
intent --> handoff : set_intent("other")

collect_info --> slot_proposal : request_slots()\n[all 6 fields collected]
collect_info --> handoff : transfer_to_human()\nor auto-trigger\n(HandoffEvaluator)

slot_proposal --> slot_proposal : get_more_slots()\n[caller wants alternatives]
slot_proposal --> confirmation : confirm_slot()
slot_proposal --> handoff : transfer_to_human()\nor auto-trigger

confirmation --> closing : send_confirmation()\n[after book_appointment]
confirmation --> slot_proposal : [caller changes mind —\nLLM returns to slot_proposal]
confirmation --> handoff : transfer_to_human()\nor auto-trigger

manage_appointment --> manage_appointment : search_patient()\n+ find_patient_appointments()\n+ select_appointment()
manage_appointment --> closing : cancel_appointment()
manage_appointment --> reschedule_slot_proposal : request_slots()\n[intent = reschedule]
manage_appointment --> handoff : transfer_to_human()\n(patient not found\nor ambiguous)

reschedule_slot_proposal --> reschedule_slot_proposal : get_more_slots()
reschedule_slot_proposal --> closing : confirm_slot()\n→ reschedule_appointment()
reschedule_slot_proposal --> handoff : transfer_to_human()

handoff --> closing : complete_handoff()

closing --> [*] : end_conversation\n(post_action)

@enduml
....

NOTE: `set_language` and `get_office_hours` are registered as *global functions* on the `FlowManager` and are therefore available in every node. `transfer_to_human` is included explicitly in `intent`, `collect_info`, `slot_proposal`, `confirmation`, `manage_appointment`, and `reschedule_slot_proposal` nodes. It must NOT be called for reschedule or cancel — those are handled autonomously by `manage_appointment`.

NOTE: The `consent` node (M2) is the pipeline's entry point. Two `pre_actions` of type `tts_say` play the greeting and the DSGVO consent question via Piper before the LLM is invoked, and `respond_immediately: False` keeps the LLM silent until the caller speaks. The first caller utterance is therefore the Ja/Nein answer, and `record_consent` is the only callable in the node. A `consent` event (session_id, timestamp, consent_given) is written to `events.jsonl` regardless of the outcome.

NOTE: "auto-trigger" refers to the `HandoffEvaluator` frame processor (see Section 9). It runs `evaluate_handoff()` on every caller transcription and calls `FlowManager.set_node_from_config(create_handoff_node())` when a regex/heuristic trigger matches — without waiting for the LLM to call `transfer_to_human`.

== 8. LLM Tool Calling

The table below lists every tool registered with the `FlowManager`. Global functions are available in all nodes; node-specific functions are injected only when that node is active.

[cols="2,2,3,2,3", options="header"]
|===
|Tool name
|Available in
|Parameters
|Returns
|Side effects on `flow_manager.state`

|`set_language`
|GLOBAL (all nodes)
|`language: "en"\|"de"` (required)
|`{language_set: str}`
|Sets `state["language"]`; queues `TTSUpdateSettingsFrame` to switch Piper voice; transitions `greeting → hours_check`

|`get_office_hours`
|GLOBAL (all nodes)
|`date: str` (ISO, optional)
|`{open, close}` or `{closed, emergency_number, message}`
|Sets `state["office_hours"]`; transitions `hours_check → intent` (open) or `hours_check → closing` (closed)

|`set_intent`
|`intent` only
|`intent: "booking"\|"reschedule"\|"cancel"\|"other"` (required)
|`{intent: str}`
|Sets `state["intent"]`; transitions `intent → collect_info` (booking), `intent → manage_appointment` (reschedule/cancel), or `intent → handoff` (other)

|`search_patient`
|`collect_info`, `manage_appointment`
|`name: str` (required), `dob: str` (optional)
|`{status: "found"\|"multiple"\|"not_found", patient?\|candidates?}`
|Sets `state["patient_record"]`; no node transition (LLM handles result inline)

|`request_slots`
|`collect_info`, `manage_appointment`
|`visit_type: enum` (required), `urgency: enum` (required), `date_range: obj` (optional)
|`{slots: [...]}`
|Sets `state["proposed_slots"]`; transitions `collect_info → slot_proposal` or `manage_appointment → reschedule_slot_proposal`

|`confirm_slot`
|`slot_proposal`, `reschedule_slot_proposal`
|`slot_id: str` (required)
|`{slot_confirmed: str}` (booking) or `{status: "rescheduled", appointment}` (reschedule)
|Booking: sets `state["chosen_slot"]`, transitions `slot_proposal → confirmation`. Reschedule: calls `reschedule_appointment()`, stores result in `state["rescheduled_appointment"]`, transitions `reschedule_slot_proposal → closing`

|`get_more_slots`
|`slot_proposal`, `reschedule_slot_proposal`
|`visit_type: enum` (required), `urgency: enum` (required), `date_range: obj` (optional)
|`{slots: [...]}`
|Updates `state["proposed_slots"]`; stays in current node

|`book_appointment`
|`confirmation` only
|`patient_id: str`, `slot_id: str`, `visit_type: enum` (all required), `notes: str` (optional)
|`{status: "confirmed"\|"slot_taken"\|"patient_not_found", confirmation_id?, appointment?}`
|On success: sets `state["confirmation_id"]` and `state["booked_appointment"]`; stays in `confirmation`

|`send_confirmation`
|`confirmation` only
|`patient_id: str` (required), `channel: "sms"\|"email"` (optional, default `sms`)
|`{status: "sent", channel, recipient, confirmation_id, timestamp}`
|Transitions `confirmation → closing`

|`find_patient_appointments`
|`manage_appointment` only
|`patient_id: str` (required)
|`{appointments: [{confirmation_id, datetime_iso, time_human, visit_type, provider}, ...]}`
|Sets `state["patient_appointments"]`; stays in `manage_appointment`

|`select_appointment`
|`manage_appointment` only
|`confirmation_id: str` (required)
|`{selected_confirmation_id: str}`
|Sets `state["selected_confirmation_id"]`; stays in `manage_appointment`

|`cancel_appointment`
|`manage_appointment` only
|`confirmation_id: str` (required)
|`{status: "cancelled"\|"not_found", appointment?}`
|On success: sets `state["cancelled_appointment"]`, transitions `manage_appointment → closing`; on not_found: stays

|`transfer_to_human`
|`intent`, `collect_info`, `slot_proposal`, `confirmation`, `manage_appointment`, `reschedule_slot_proposal`
|`reason: enum` (required): `caller_requested\|medical_question\|billing_dispute\|ambiguous_patient\|outside_scope\|frustration`
|`{handoff: true}`
|Sets `state["handoff_reason"]`; transitions current node → `handoff`. Must NOT be called for reschedule or cancel requests.

|`complete_handoff`
|`handoff` only
|_(none)_
|`{done: true}`
|Transitions `handoff → closing`
|===

NOTE: `reschedule_appointment` is invoked by the `confirm_slot` handler in `reschedule_slot_proposal`, not exposed to the LLM directly, so the LLM only ever picks a slot — it does not need to construct the appointment-move call.

== 9. Handoff Triggers

`evaluate_handoff()` in `handoff.py` implements five trigger types using regex patterns and simple heuristics. It is a pure Python function with no Pipecat dependencies.

It is wired into the pipeline via the `HandoffEvaluator` frame processor in `processors.py`, which sits between the STT service and the user-side context aggregator. On every caller `TranscriptionFrame` it calls `evaluate_handoff(state, text, language_probability)`; if a reason is returned it sets `flow_manager.state["handoff_reason"]` and calls `flow_manager.set_node_from_config(create_handoff_node())`, forcing a transition regardless of what the LLM would have decided.

Handoff is gated to the `intent`, `collect_info`, `slot_proposal`, `confirmation`, `manage_appointment`, and `reschedule_slot_proposal` nodes — greeting, hours_check, handoff and closing are excluded.

IMPORTANT: reschedule and cancel requests are NOT handoff triggers. They are handled autonomously by the `manage_appointment` flow (section 4). A previous version of the evaluator matched on "reschedule"/"absagen"/"verschieben" — that behaviour was removed when the self-serve flow was added.

The `language_probability` input is sourced from `WhisperSTTWithConfidence` (subclass of `WhisperSTTService` in `processors.py`) which captures faster-whisper's `info.language_probability` and stashes it in `TranscriptionFrame.result` — the built-in service discards this value.

[cols="2,3,3", options="header"]
|===
|Trigger type
|Condition
|Example patterns

|`CALLER_REQUESTED`
|Regex match on explicit human-request phrases (EN + DE)
|"speak to a person", "transfer me", "Mensch sprechen", "Mitarbeiter", "verbinden"

|`MEDICAL_QUESTION`
|Regex match on symptom/treatment/medication content
|"symptoms", "antibiotic", "Entzündung", "Antibiotika", "Behandlung empfehlen"

|`BILLING_DISPUTE`
|Regex match on invoice/insurance dispute patterns
|"invoice", "overcharged", "Rechnung", "Kostenübernahme", "Beschwerde"

|`LOW_STT_CONFIDENCE`
|Whisper `language_probability` < 0.45 for 2+ consecutive turns
|(`state["stt_low_confidence_count"]` ≥ `low_confidence_consecutive_turns` in settings.yaml)

|`FRUSTRATION`
|Jaccard similarity ≥ 0.6 between consecutive utterances, count ≥ 2
|Caller repeating nearly the same sentence twice in a row
|===

=== 10.2 office_hours.yaml

The file is loaded by `tools/pms_mock.py:get_office_hours()`. Structure:

[source,yaml]
----
timezone: "Europe/Berlin"

weekday_hours:
  monday:    { open: "08:00", close: "18:00" }
  # ... tuesday–friday; saturday/sunday: closed

exceptions:                      # date-specific overrides
  "2026-12-24": { open: "08:00", close: "12:00" }
  "2026-05-01": closed

holidays:                        # full closures — list of ISO date strings
  - "2026-01-01"                 # DE: Neujahr / US: New Year's Day
  # ... DE public holidays + US federal holidays

after_hours_routing:
  emergency_number_de: "+49-800-111-2222"
  emergency_number_us: "+1-800-555-0123"
  voicemail_enabled: true
  callback_logging: true
  message_de: "Unsere Praxis ist derzeit nicht erreichbar. ..."
  message_en: "Our practice is currently closed. ..."
----

Priority order for `_is_office_open()`: *exceptions* > *holidays* > *weekday_hours*.

== 12. Known Limitations & Roadmap

=== 12.1 Current Limitations (Phase 0)

[cols="1,3", options="header"]
|===
|Area
|Description

|Handoff routing
|`evaluate_handoff()` is now wired into the pipeline via `HandoffEvaluator` (see Section 9). Both LLM-driven handoffs (`transfer_to_human` tool call) and regex/heuristic-driven auto-handoffs fire. AMBIGUOUS_PATIENT is still routed only through the LLM's `transfer_to_human` call in `collect_info` — no pure-Python evaluator for it yet.

|Context window
|Conversation history grows unbounded within a session. There is no context trimming or summarization. Long calls will eventually exceed the LLM's context limit.

|Concurrency
|The FastAPI app spawns one Pipecat pipeline per WebRTC connection (`asyncio.create_task`). There is no shared state, no call queue, and no concurrency cap. Capacity is limited by available CPU/RAM.

|PMS data
|`tools/pms_mock.py` uses in-memory dicts. All data (patients, bookings) resets on server restart. Not suitable for production without replacing with a real PMS adapter.

|Authentication
|No caller identity verification. Any browser can connect. No PIN, no callback verification, no fraud detection.

|Debug logging
|`DebugFrameLogger` is inserted into the pipeline unconditionally (always on). It prints every VAD, STT, LLM, and TTS event to stdout with ANSI colour codes. Remove or gate behind a flag before production.
|===

== 1. Overview

The Dental Office Voice Receptionist is a voice-first appointment booking agent for a dental practice. Callers interact with the agent through spoken natural language â€” the system handles greeting, language selection, patient lookup, slot proposal, booking confirmation, and graceful escalation to a human receptionist.

*Phase 0 (this repo)* is a proof-of-concept that runs entirely on localhost. A browser-based WebRTC client (`static/index.html`) replaces a real telephone connection. No external telephony account or cloud subscription is required â€” the only optional external dependency is the Groq API for LLM inference; the system also runs fully offline with Ollama.

*Supported locales:* The agent speaks German (formal _Sie_-form) or English. The active locale is set once at startup via the `OFFICE_LOCALE` environment variable (`de` or anything else â†’ `en`). Language is locked for the entire call â€” there is no mid-call detection. The initial greeting, TTS voice, and after-hours routing message are all derived from this value.

The pipeline is built on *Pipecat 1.0.0* with the `pipecat-flows` library managing the conversation state machine. The LLM backend is configurable: either *Groq* (`llama-3.3-70b-versatile`, cloud) or *Ollama* (`qwen2.5:14b`, local). Both are exposed through the OpenAI-compatible `OpenAILLMService`. Speech recognition uses *faster-whisper* (multilingual EN/DE). Text-to-speech uses *Piper* with locally cached `.onnx` model files.

== 4. Conversation State Machine

The flow is managed by *pipecat-flows* `FlowManager`. Each state is a `NodeConfig` with a set of callable functions; state transitions happen exclusively via tool call return values from the handlers in `flows/nodes.py`.

== 9. Handoff Triggers

`evaluate_handoff()` in `handoff.py` implements six trigger types using regex patterns and simple heuristics. It is a pure Python function with no Pipecat dependencies.

IMPORTANT: `evaluate_handoff()` is currently *not wired into the pipeline*. In Phase 0 the LLM routes to the handoff node exclusively via the `transfer_to_human` tool call. `evaluate_handoff()` is defined and tested but not called from any pipeline frame processor.

[cols="2,3,3", options="header"]
|===
|Trigger type
|Condition
|Example patterns

|`CALLER_REQUESTED`
|Regex match on explicit human-request phrases (EN + DE)
|"speak to a person", "transfer me", "Mensch sprechen", "Mitarbeiter", "verbinden"

|`MEDICAL_QUESTION`
|Regex match on symptom/treatment/medication content
|"symptoms", "antibiotic", "EntzĂĽndung", "Antibiotika", "Behandlung empfehlen"

|`BILLING_DISPUTE`
|Regex match on invoice/insurance dispute patterns
|"invoice", "overcharged", "Rechnung", "KostenĂĽbernahme", "Beschwerde"

|`RESCHEDULE`
|Regex match on reschedule/cancel patterns
|"reschedule", "cancel", "Umtermin", "Termin absagen", "kann nicht kommen"

|`LOW_STT_CONFIDENCE`
|Whisper `language_probability` < 0.45 for 2+ consecutive turns
|(`state["stt_low_confidence_count"]` â‰Ą `low_confidence_consecutive_turns` in settings.yaml)

|`FRUSTRATION`
|Jaccard similarity â‰Ą 0.6 between consecutive utterances, count â‰Ą 2
|Caller repeating nearly the same sentence twice in a row
|===

== 12. Known Limitations & Roadmap

=== 12.1 Current Limitations (Phase 0)

[cols="1,3", options="header"]
|===
|Area
|Description

|Handoff routing
|`evaluate_handoff()` in `handoff.py` is fully implemented (regex + heuristics) but *not connected* to the Pipecat pipeline. The LLM routes to handoff exclusively via the `transfer_to_human` tool. Consequence: a caller who does not explicitly ask for a transfer but triggers a FRUSTRATION or LOW_STT_CONFIDENCE pattern will not be auto-escalated in Phase 0.

|Context window
|Conversation history grows unbounded within a session. There is no context trimming or summarization. Long calls will eventually exceed the LLM's context limit.

|Concurrency
|The FastAPI app spawns one Pipecat pipeline per WebRTC connection (`asyncio.create_task`). There is no shared state, no call queue, and no concurrency cap. Capacity is limited by available CPU/RAM.

|PMS data
|`tools/pms_mock.py` uses in-memory dicts. All data (patients, bookings) resets on server restart. Not suitable for production without replacing with a real PMS adapter.

|Authentication
|No caller identity verification. Any browser can connect. No PIN, no callback verification, no fraud detection.

|Debug logging
|`DebugFrameLogger` is inserted into the pipeline unconditionally (always on). It prints every VAD, STT, LLM, and TTS event to stdout with ANSI colour codes. Remove or gate behind a flag before production.
|===
